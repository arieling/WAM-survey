## One-Sentence Summary

> Im2Flow2Act uses object-centric flow (generated by a Stable Diffusion + AnimateDiff network from human or language-conditioned demonstrations) as a cross-domain manipulation interface, then extracts actions via a diffusion-based policy conditioned on temporally-aligned latent flow — achieving 81% average real-world success across rigid, articulated, and deformable objects without any robot training data.

---

## Core Contributions

1. **Object-Centric Flow as Cross-Domain Interface**: Rather than capturing full-scene or embodiment-specific motion, Im2Flow2Act generates object-only flow (N keypoints on the target object), eliminating the embodiment distribution shift that plagues grid-flow methods requiring robot training data.
2. **Flow Generation from Human Demonstrations**: Stable Diffusion U-Net + AnimateDiff motion modules generate plausible object flow from initial frames conditioned on human video demonstrations or language instructions — without robot data.
3. **Temporal Alignment Module**: Transformer encoder predicts remaining task flow in latent space from current state, bridging the gap between offline-generated complete flow and online robot execution at each timestep.
4. **Sim-to-Real Cross-Domain Policy**: Policy trained entirely on MuJoCo simulated exploration data (4,800 trajectories, Bézier-curve primitives) can execute real-world manipulation without real robot demonstrations.

---

## Problem Background

### Problem to Solve
Robot manipulation policies require large amounts of robot demonstration data, but robot data is expensive to collect. Human manipulation videos are abundant but cannot be directly used due to embodiment differences. How can human video demonstrations and simulated robot exploration data be combined to enable real-world robot manipulation without robot training data?

### Limitations of Existing Methods
- ATM (Any-point Tracking): uses grid flow capturing both object and embodiment motion — requires real robot training data because robot hand/arm motion is included in the flow representation.
- Heuristic policies (RANSAC-based): fail on deformable objects; require geometry assumptions.
- Language-conditioned VLAs: require robot demonstration data; poor generalization to novel object types.

### Motivation
Object-centric flow captures only target object motion — no embodiment motion. This makes the flow representation domain-invariant: the same object flow pattern appears in human videos, simulation, and real-robot execution. A policy trained on simulated exploration can follow flows generated from human demonstrations at deployment time.

## Method Details

### Model Architecture

Im2Flow2Act has **two main components**:

1. **Flow Generation Network**: SD U-Net + AnimateDiff → object-centric flow video from initial frame + demonstration/language
2. **Flow-Conditioned Policy**: Temporal alignment transformer + diffusion-based action policy

### Core Modules

#### Module 1: Object-Centric Flow Generation

**Design Motivation**: Object-centric flow (tracking only object keypoints, not the robot hand or background) eliminates embodiment-specific information — the same flow pattern is achievable by any robot embodiment.

**Implementation**:
- Grounding DINO detects object bounding boxes in the initial frame
- N keypoints sampled uniformly within the bounding box → represented as rectangular flow image (shape: $3 \times H \times W$, channels: x-displacement, y-displacement, keypoint mask)
- Stable Diffusion autoencoder: 8× spatial downsampling to latent space
- AnimateDiff motion modules: self-attention across temporal dimension for temporal coherence
- SD U-Net backbone: pretrained (frozen encoder, fine-tuned decoder with LoRA)
- Motion modules: trained from scratch
- Conditioned on: initial frame + human demonstration video frames or language instruction

#### Module 2: State Encoder

**Design Motivation**: The policy must understand the current state of keypoint positions in 3D space to compute the remaining task flow and select appropriate actions.

The state at timestep $t$ is encoded as:

$$s_t = \phi(f_t, x_0)$$

where $f_t$ denotes the N keypoint 2D positions tracked in the current frame, $x_0$ is the initial 3D keypoint coordinates from the depth sensor at the initial frame, and $\phi$ is a transformer encoder with 2D sinusoidal positional encoding on keypoint locations. The initial 3D coordinates provide depth context that is especially important for language-conditioned settings without human demonstration depth.

#### Module 3: Temporal Alignment Module

**Design Motivation**: Complete task flow is generated offline (before execution), but the policy executes step-by-step. A temporal alignment module predicts "remaining flow" from the current state, synchronizing offline flow plan with online execution.

**Implementation**:
- Transformer encoder: predicts remaining task flow $\hat z_t$ in latent space from current state $s_t$ and full task flow latent $z$

The temporal alignment is trained with an L2 loss in latent space that uses a stop-gradient on the prediction to prevent trivial collapse:

$$\mathcal L_{align} = \|\hat z_t^{\text{detach}} - z_t\|^2$$

where $\hat z_t$ is the temporal alignment module's predicted remaining flow and $z_t$ is the ground-truth remaining flow latent at the current execution timestep. At inference, the predicted $\hat z_t$ conditions the diffusion policy at each step.

#### Module 4: Diffusion-Based Action Policy

**Design Motivation**: Diffusion policy handles multi-modal action distributions arising from diverse manipulation strategies; conditioning on latent remaining flow provides geometric guidance for action selection.

**Implementation**:

The action distribution at each timestep is modeled as:

$$p(a_t \mid z_t, s_t, \rho_t)$$

where $z_t$ is the latent remaining flow from the temporal alignment module, $s_t$ is the state representation, and $\rho_t$ is the robot proprioception. The policy outputs a 6-DOF end-effector action + gripper state as a chunk of length $L$.

- Trained on simulated exploration data only (4,800 MuJoCo trajectories with Bézier-curve primitives)

## Experiments

### Datasets

| Dataset | Scale | Characteristics | Usage |
|---------|-------|----------------|-------|
| MuJoCo exploration | 4,800 trajectories | Rigid, articulated, deformable; Bézier-curve primitives; UR5e | Policy training (sim) |
| Human demo videos | 4 tasks | Pick-place, pouring, drawer, cloth folding; 30 FPS | Flow generation conditioning |

### Simulation Results

Im2Flow2Act substantially outperforms both ATM (which requires real robot data) and GridFlow (which captures embodiment-specific motion) in simulation:

| Task | Im2Flow2Act | ATM | GridFlow |
|------|------------|-----|---------|
| Demo-Cond Pick&Place | 100% | — | 30% |
| Demo-Cond Pouring | 95% | — | 25% |
| Language-Cond Pick&Place | 90% | 50% | — |

Im2Flow2Act outperforms ATM by +40% on language-conditioned tasks; outperforms GridFlow by +65–70% on demo-conditioned tasks.

### Real-World Results

The pipeline is evaluated without any real robot training data — the policy is trained entirely in MuJoCo simulation:

| Task | Im2Flow2Act | Heuristic |
|------|------------|-----------|
| Demo-Cond Pick&Place | 95% | 70% |
| Language-Cond Pick&Place | 90% | — |

+25% over heuristic policy on pick&place; 81% average across all real-world tasks (rigid, articulated, deformable).

### Ablation Results

The ablation below highlights the critical role of each component. Temporal alignment is the single most important module — its removal causes a 35–40% collapse in real-world performance:

| Configuration | Sim→Real Drop |
|--------------|---------------|
| Without temporal alignment | −35–40% |
| Grid flow (vs. object flow) | −30% |
| Without initial 3D keypoints | −5–10% |
| Without pretrained SD | slower convergence |

Object-centric flow provides +30% over grid flow on cross-embodiment tasks.

### Implementation Details

- **Flow Generator Backbone**: Stable Diffusion U-Net (pretrained frozen encoder, fine-tuned decoder + LoRA)
- **Motion Modules**: AnimateDiff (trained from scratch, temporal self-attention)
- **Flow Representation**: $3 \times H \times W$ rectangular image (x-displacement, y-displacement, mask)
- **Latent Compression**: SD autoencoder, 8× spatial downsampling
- **Object Detection**: Grounding DINO (frozen, zero-shot)
- **Policy**: Diffusion Policy (6-DOF end-effector + gripper, chunk length $L$)
- **Temporal Sampling**: Random T-frame sampling from episodes during training
- **Robot (Sim)**: UR5e (MuJoCo)
- **Control Mode**: 6-DOF end-effector
- **Tasks**: Pick-place, pouring, drawer opening, cloth folding (rigid, articulated, deformable)

## Critical Analysis

### Strengths
1. No robot training data required — policy trained entirely in simulation, flow generated from human videos.
2. Object-centric flow elegantly eliminates embodiment distribution shift — principled and general.
3. 81% real-world success across rigid, articulated, and deformable objects demonstrates broad manipulation capability.

### Limitations
1. **Keypoint tracking failures**: Object-centric flow requires reliable keypoint detection; occlusion and similar-appearance distractors degrade tracking.
2. **15% sim-to-real drop**: Residual performance gap from simulation to real-world indicates incomplete domain transfer.
3. **Flow generation quality**: SD+AnimateDiff may generate physically implausible flows for complex interactions.
4. **Single-object focus**: Object-centric paradigm struggles with multi-object interaction tasks.

### Potential Improvements
1. Closed-loop flow replanning from current observation for drift correction.
2. Multi-object flow for manipulation tasks involving tool use or hand-offs.
3. Better sim-to-real transfer with domain randomization in flow generation.
